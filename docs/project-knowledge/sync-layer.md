# GHL ↔ Xero Sync Layer

## Status: LIVE (deployed 3 March 2026)

## Purpose
Universal link between GHL (CRM), Supabase (database), and Xero (accounting) using type-prefixed job numbers. Replaces broken name-based matching (was 22% match rate, now 65%+). Non-destructive — runs alongside Tradify until fully phased out.

## Architecture Flow
```
Lead comes in → GHL Contact exists
    ↓
Scope complete (ghl-proxy link action) fires automatically:
    1. Generate job number: SWF-25001
    2. Create/find Xero contact (by email, then name)
    3. Push monetary value to GHL opportunity
    ↓
Quote sent (send-quote) → also pushes $ to GHL
    ↓
Bookkeeper creates invoice in Xero:
    - Contact already exists (auto-created)
    - Puts SW number in Reference field
    ↓
Invoice sync (xero-sync) → sees Reference "SWF-25001" → auto-links to job
    ↓
Dashboard shows accurate per-job revenue attribution
```

## Job Number Format
- **SWP-25001** = Patio
- **SWF-25002** = Fencing
- **SWD-25003** = Decking
- **SWR-25004** = Renovation or Roofing
- **SWI-25005** = Insurance
- **SW-25006** = Other/unknown type
- Sequence starts at 25000 (above existing Tradify max ~SW23324)
- Generated by `next_job_number(job_type)` Postgres function
- Assigned at scope complete, NOT at lead creation (avoids wasting numbers)

## Database Changes (Migration 012)
- `job_number_seq` sequence starting at 25000
- `jobs.job_number` column (unique index where not null)
- `jobs.xero_contact_id` column
- `jobs.xero_quote_id` column
- `next_job_number(job_type)` SQL function
- `jobs.type` constraint expanded: fencing, patio, combo, decking, renovation, insurance, roofing

## Edge Function Changes

### xero-sync (new actions)
- `create_or_find_contact` (POST) — searches Xero by email then exact name, creates if not found, updates jobs.xero_contact_id, upserts contact_matches, links existing invoices by xero_contact_id
- `match_invoices_by_reference` — finds xero_invoices with SW refs and no job_id, links them
- `backfill_xero_contacts` — processes active jobs without xero_contact_id in batches (?limit=10). Rate-limited for Xero API (pause every 5 jobs)
- `xeroPost()` helper added — POST/PUT to Xero API with rate-limit retry (same pattern as xeroGet)
- `syncInvoices` enhanced — after each invoice upsert, checks Reference for SW number, auto-links to job
- `matchContacts` improved — normalizeName() strips Mr/Mrs/Dr, surname+initial matching, logs unmatched to webhook_log

### ghl-proxy (link action updated)
After existing scope complete logic, adds 3 non-blocking steps:
1. `sb.rpc('next_job_number', { job_type })` → update job
2. Internal fetch to `xero-sync?action=create_or_find_contact`
3. GHL API PUT to update opportunity monetaryValue
Enhanced response includes: jobNumber, xeroContact, monetaryValue

### reporting-api (matchInvoicesToJobs updated)
- Added Pass 0: reference matching (before Pass 1 exact name, Pass 2 surname+initial)
- Pattern: `/SW[A-Z]?-?(\d{3,5})/i` on invoice reference field
- Builds jobByNumber lookup from jobs with job_number
- data_quality updated: contact_match_rate_low = false, CLV data_quality = 'moderate'

### send-quote (GHL push added)
- After quote send + job status update, pushes monetaryValue to GHL opportunity (non-blocking)

## Known Limitations
- **Name matching is exact** — "Brett Hunt" won't match "Brett and Steph Hunt". Email is the reliable path. Client email in scoping tool is important.
- **Xero rate limit** — 60 requests/minute. Backfills must be batched (?limit=10)
- **WORKER_LIMIT** — Supabase Edge Functions have compute limits. Backfills timeout if batch too large.
- **Supabase PostgrestFilterBuilder** does NOT have `.catch()` — must use try/catch blocks, not .catch()
- **Channel attribution mostly empty** — "Unknown"/"Unattributed" because GCLID/lead_source not flowing through contact_matches properly
- **Sequence values consumed on test** — SWP-25000, SWF-25001, SWP-25002, SWF-25003 already used (first 2 from initial test, last 2 from verification test). Next real job number will be SWP-25004+
